## 1. 核心治理原则
*   **代码仓库是唯一事实来源**：严禁仅依赖会话上下文。所有意图、决策、构建步骤和进度必须实时记录在物理文档中。**每次会话唤醒时，Agent 必须优先读取核心文档对齐状态。**
*   **全路径验证高于单元测试**：任务完成的唯一标准是跑通 PRD 中定义的完整业务流程。跑通完整流程才算真正验证。
*   **动态任务管理与 Context 瘦身**：任务清单必须保持“聚焦执行”状态，主任务与子任务分层展示。严禁平铺导致的视觉混乱，**严禁将大段报错日志（>50行）直接写入文档**。

## 2. 信息分流决策树 (路由规则)
遇到变动或新知识时，Agent 必须按下表逻辑准确分流信息，严禁胡乱堆放：

| 信息类型 | 对应文档 | 包含内容示例 |
| :--- | :--- | :--- |
| **业务意图 (做什么？)** | **PRD.md** | 需求变更、验收标准 (Acceptance Criteria) 调整。 |
| **技术知识 (怎么做/为什么？)** | **DESIGN.md** | 架构选型、接口定义、**踩坑复盘与解决方案**。 |
| **当前状态 (进度如何？)** | **TASKS.md** | 层级化任务清单、执行障碍 (Obstacles)、**报错摘要**。 |
| **质量证据 (结果对吗？)** | **TEST_REPORT.md** | 单元测试结论、全路径业务验证证据。 |

## 3. 层级化任务管理规则 (TASKS.md)
*   **一级清单 (Global Goals)**：列出项目初期的所有主任务。
*   **二级/三级清单 (Focused Sub-tasks)**：仅针对当前正在执行的主任务进行动态拆解。
*   **状态符号规范**：` [ ] ` 待处理, ` [/] ` 进行中, ` [x] ` 已完成, ` [!] ` 遇到障碍。
*   **清理与折叠**：当主任务完成并进入下一项时，需将已完成的详细子任务进行“逻辑折叠”或移除，只保留主任务的 ` [x] ` 状态；长篇报错堆栈只能保存在本地 `.log` 临时文件中，TASKS.md 仅保留一句话报错摘要。

## 4. 标准作业流程 (SOP)
*   **Phase 0：环境唤醒**：读取 TASKS.md 判断当前进度，读取 PRD.md 对齐本次目标。
*   **Phase 1：初始化**：运行环境检查指令（如 init.sh），确保工作空间就绪。
*   **Phase 2：需求对齐**：在 PRD.md 中明确“全路径验证”成功的具体表现和验证手段。
*   **Phase 3：双层 TDD**：先保单元层逻辑正确，再保业务层流程跑通。若遇阻，将摘要录入 TASKS.md。若解决重大 Bug，必须将经验沉淀至 DESIGN.md。
*   **Phase 4：审计交付**：只有通过“全路径验证”，产出 TEST_REPORT.md，**并执行代码 Commit 后**，方可宣告当前节点完成。
*   **Phase 5：清理与交接**：：会话结束前更新 progress.md，记录当前精确位置与避坑经验

## 5. 兜底规则
*   **分类疑难**：若不确定信息分类，先完整记入 TASKS.md 的 Obstacles 区域并请求人工指令。
*   **禁止假性宣告**：严禁在未更新相关文档、未通过全路径验证的情况下宣告任务完成。
