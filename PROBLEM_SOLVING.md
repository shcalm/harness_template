# PROBLEM_SOLVING.md：问题分析与解决流程

> 本文件约束**调试方法**（思维流程、判断标准），不约束**文档操作**（写哪份、写什么字段）。
> 文档落盘按 AGENTS.md §7；验证记录格式按 AGENTS.md §5；启用条件由本文件定义。

## 何时启用本流程

**简单问题直接处理，不需要本流程。** 以下特征说明问题足够简单，LLM 可以直接给出答案：

* 报错信息明确指向单一原因
* 影响范围局限在单个函数或模块内
* 原因和修复方式一眼可见

**复杂问题必须启用本流程。** 出现以下任一特征，跳过流程容易跑偏：

| 特征 | 说明 |
|:---|:---|
| 跨模块/跨边界 | 问题涉及多个模块的交互，根因不在单一位置 |
| 现象与原因不对应 | 报错位置和实际根因不在同一个地方 |
| 间歇性/难复现 | 不能稳定触发，依赖时序、并发或外部状态 |
| 修了又出现 | 之前"修好"过，说明根因未真正确认 |
| 影响范围不清楚 | 不确定改动会影响哪些模块 |
| 假设多于证据 | 推断链超过两步，每步都是猜测 |

---

## 核心原则

证据先于结论。从整体架构把握问题，确认边界，再向内定位根因。局部正确不等于系统正确。

---

## 解决问题的标准流程

### 第 1 步：定义问题

用一句话写清楚：

| 字段 | 内容 |
|:---|:---|
| 现象 | 观察到了什么（报错/异常行为/性能下降） |
| 触发条件 | 什么操作或输入导致了这个现象 |
| 影响范围 | 哪些功能、用户、环境受影响 |
| 预期行为 | 正确的结果应该是什么 |

> 如果这四个字段填不完整，先补信息再继续。

---

### 第 2 步：定位架构边界

**先看全局，再看细节。** 在翻代码之前，先回答以下问题：

```
1. 系统由哪些模块组成？各模块的职责边界是什么？
2. 问题路径上涉及哪些模块？调用链是什么？
3. 问题发生在哪个边界？
   - 模块内部逻辑？
   - 模块间接口（调用方/被调用方的契约）？
   - 外部依赖边界（DB、第三方服务、OS）？
4. 哪一侧的行为是"可信的"？哪一侧是"可疑的"？
```

画出或写出问题路径：

```
[调用方] → [模块 A] → [模块 B] → [外部服务]
                         ↑
                    问题发生在这里？
```

> 跳过这一步直接看代码，会陷入局部正确但系统错误的盲区。边界不清楚时，先读 `DESIGN.md` 架构概览和接口契约。

---

### 第 3 步：收集证据

按优先级依次收集：

1. **复现** — 能稳定复现是最强证据，写出最小复现步骤。
2. **日志** — 找到报错时间点前后的完整日志，覆盖边界两侧。
3. **环境信息** — OS、运行时版本、关键配置、最近变更记录。
4. **对照** — 找一个"正常"的对照案例，对比差异。

> 收集日志时要覆盖**边界两侧**：调用方发出了什么，被调用方收到了什么，两者是否一致。

---

### 第 4 步：缩小范围

用排除法，每次排除一个维度：

```
是环境问题？       → 换环境复现 → 能/不能
是输入问题？       → 换最小输入复现 → 能/不能
是最近变更引入的？ → git bisect 或回滚验证 → 能/不能
是依赖版本问题？   → 锁定版本验证 → 能/不能
是边界契约不一致？ → 对比接口定义与实际传参 → 一致/不一致
```

每次排除都写结论，不要靠记忆。

---

### 第 5 步：确认根因

根因标准：能解释所有已知现象，且找到代码/配置中的具体位置。

| 判断项 | 说明 |
|:---|:---|
| 能解释全部现象 | 包括间歇性的和边界情况 |
| 有代码/配置定位 | 具体到文件、行号或配置键 |
| 边界归属明确 | 是调用方的问题、被调用方的问题，还是契约本身的问题 |
| 能预测修复效果 | 修改后现象应消失 |

> 如果只能解释部分现象，说明还有第二个根因或假设不成立，继续收集证据。

---

### 第 6 步：制定修复方案

写出至少两个方案再选择：

| 方案 | 描述 | 优点 | 代价/风险 |
|:---|:---|:---|:---|
| 方案 A | `[最小改动]` | 风险低，可快速验证 | `[局限性]` |
| 方案 B | `[彻底修复]` | 根治问题 | 改动范围大，需更多验证 |

选择标准：优先选最小改动能验证根因的方案；根治方案另开任务。

---

### 第 7 步：实施与验证

```
1. 实施最小改动
2. 复现步骤重跑 → 现象消失？
3. 原有测试全部通过？
4. 边界两侧行为验证（调用方/被调用方输入输出是否符合契约）
5. 边界情况验证（空输入、并发、异常路径）
```

验证记录格式按 AGENTS.md §5。

---

### 第 8 步：归档

根因确认且修复验证通过后，按 AGENTS.md §7 文档更新时机执行落盘。本流程到此结束。

---

## 遇到卡住的情况

| 卡住原因 | 处理方式 |
|:---|:---|
| 边界不清楚 | 回到 `DESIGN.md` 架构概览，补充模块关系图，再继续 |
| 无法复现 | 增加日志覆盖边界两侧，等待下次触发，记录"待观察" |
| 日志不足 | 修改代码加日志，重新触发，再分析 |
| 根因指向外部依赖 | 最小化复现后提 issue 或写规避方案，记录 DEBT-ID |
| 修复引入新问题 | 回滚，重新从第 2 步检查边界归属 |
